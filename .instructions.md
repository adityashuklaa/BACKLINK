# VestaCall Backlink Automation — Agent Challenge & Optimization Protocol

**Version 1.0** | Workspace: `c:\Users\Dev\Desktop\backlink`  
**Purpose**: Continuously challenge code quality, efficiency, and output to maximize backlink generation while avoiding detection and managing team productivity.

---

## I. Core Project Principles

### 1. **Detection Avoidance First**
- Every submission must include randomized delays (3–8s minimum between actions).
- Browser automation must use Playwright with anti-detection headers and user-agent rotation.
- HTTP requests must vary user-agents and include realistic referers.
- **If a change removes, reduces, or bypasses rate limiting or randomization, it is CRITICAL and must be rejected.**

### 2. **Backlink Quality > Quantity**
- Prioritize high-authority sites (Crunchbase, G2, BBB, Yellow Pages, review platforms).
- Every submission MUST be logged to CSV with status, URL, and timestamp.
- Do not retry failed submissions endlessly; investigate root cause (detection, rate limit, auth failure) before resubmitting.
- Success rate target: ≥75% across all strategies.

### 3. **Audit Trail & Deduplication**
- All backlink submissions logged to `output/backlinks_log.csv` with columns: timestamp, strategy, site_name, submission_url, backlink_url, status, notes.
- CSV logger prevents duplicate submissions by checking site_name before action.
- Any strategy that skips CSV logging is incomplete.

### 4. **Team Accountability & Transparency**
- Each strategy is owned by one person but accountable to the team.
- Submissions are tracked per person and per strategy in efficiency metrics.
- Blockers and risks are surfaced immediately (e.g., "Yelp verification required—blocking submissions").
- Success/failure reasons are explicit in CSV notes field.

## II. Best Practices Checklist for Every Strategy

### For each Python strategy file (`strategies/*.py`):

**A. Logging & Observability**
- [ ] Strategy function has docstring explaining target site, submission method, expected rate limit.
- [ ] Every site submission attempt is logged to CSV (success, skip, failure).
- [ ] Failure reasons are explicit in `notes` column (e.g., "Rate limited", "Auth required", "Already submitted").
- [ ] Progress printed to console every N submissions (N ≤ 5).
- [ ] Execution time logged; if exceeds expected baseline (see section IV), alert logged.

**B. Rate Limiting & Detection Avoidance**
- [ ] Random delay between 3–8 seconds between submissions (use `core.rate_limiter`).
- [ ] No two strategies run in parallel without separate browser contexts.
- [ ] User-agent rotated per request (in HTTP client and Playwright).
- [ ] Referer header included in all HTTP requests (realistic source URL).
- [ ] If strategy uses Playwright, browser launched in non-headless mode or with stealth flags.
- [ ] Cumulative delays recorded; if strategy exceeds 10% overhead, flagged for optimization.

**C. Error Handling & Recovery**
- [ ] Network errors caught and logged (timeout, DNS, SSL) with retry count.
- [ ] Auth failures logged separately from submission failures; no silent re-auth.
- [ ] If ≥3 consecutive failures on same site, strategy pauses and alerts user.
- [ ] Dry-run mode available (print submissions without submitting).

**D. Data Integrity**
- [ ] All user input (emails, site URLs, post titles) validated before submission.
- [ ] Config values checked at startup (SMTP creds, browser path, API keys); exit if missing.
- [ ] Pre-check: CSV loader verifies no duplicate submissions today (same site, same strategy).

## III. Code Quality Standards

### All Python Files (`core/*.py`, `strategies/*.py`, `main.py`)

**Naming & Documentation**
- [ ] Functions > 5 lines have docstrings (what, why, args, returns, raises).
- [ ] Variable names are descriptive; no single-letter vars except loops (i, j, k).
- [ ] Constants in UPPER_SNAKE_CASE; magic numbers documented.
- [ ] Module-level docstring explains module purpose and public API.

**Type Hints & Validation**
- [ ] Function signatures include type hints for args and return (e.g., `def submit(url: str, retry: int = 3) -> bool:`).
- [ ] Input validation at function entry (not in caller).
- [ ] Return types meaningful (bool, dict, List[str], not generic Any).

**Exception Handling**
- [ ] Specific exceptions caught (not bare `except Exception`).
- [ ] Custom exceptions defined in `core/exceptions.py` for domain-specific errors.
- [ ] Errors include context (e.g., `raise RateLimitExceeded(f"Waiting {retry_after}s before retry")`).

**DRY & Reusability**
- [ ] Common logic extracted to `core/` utilities (HTTP client, browser, rate limiter, CSV logger).
- [ ] Strategies use shared utilities; duplicated code flagged.
- [ ] Config loaded once in `main.py`, passed as args to strategies (not re-loaded per strategy).

**Testing & Validation**
- [ ] Dry-run mode available in every strategy (preview submissions without sending).
- [ ] CSV log entries manually spot-checked before first run (verify column order, timestamps, URLs).

## IV. Performance Baselines & Expectations

**Strategy Runtime Targets** (single machine, single browser):
- **Directories**: ~3–5 min for 20 sites (9–15s per submission, includes rate limiting).
- **Social Bookmarking**: ~5–8 min for 8 platforms (30–60s per platform, includes browser load/auth).
- **Forum Profiles**: ~10–15 min for 5+ forums (2–3 min per forum, includes registration + anti-captcha delays).
- **Guest Post Outreach**: ~8–12 min for 20 emails (24–36s per email, includes email send + SMTP delays).
- **Blog Comments**: ~6–10 min for 10 blogs (36–60s per blog, includes browser navigation + CAPTCHA).

**Total Sequential Run**: ~45–75 min (no parallelization yet).

**Optimization Targets**:
- Target: Reduce sequential runtime by 20–30% via parallelization (2–3 concurrent strategies, shared browser context pool).
- CSV logging overhead: <5% of total runtime.
- Rate limiter randomization cost: Acceptable; do not optimize away.

## V. Detection Avoidance Guidelines

### High-Risk Indicators (Flag Immediately)
1. ❌ Submitting >10 sites/hour on same IP to same domain → **Leads to IP ban**.
2. ❌ Identical user-agent across all requests → **Signature of bot**.
3. ❌ Submitting at exact 5s intervals → **bot-like pattern; must randomize 3–8s**.
4. ❌ No referer header → **Suspicious; include realistic source**.
5. ❌ Headless browser without anti-detection flags → **Easy to detect**.
6. ❌ Reusing browser context across unrelated submission batches → **Cookies leak identity**.

### Mitigations (Non-Negotiable)
- [ ] Random delays 3–8s between submissions (implement in `core.rate_limiter`).
- [ ] Browser rotation per session or daily (or use proxy if available).
- [ ] User-agent rotation per request (from `core.http_client`).
- [ ] Referer header matches target domain (e.g., submit to Reddit → referer = reddit.com).
- [ ] Playwright browser launched with anti-detection plugin or headless=False.
- [ ] If rate-limited (HTTP 429, timeout), pause strategy for 60s min, log incident, continue.

## VI. Team Efficiency Metrics & Tracking

### Per-Strategy Metrics (tracked weekly)
- **Submissions/Person/Week**: Total backlinks submitted by person.
- **Success Rate**: % of submissions with status='success' (target ≥75%).
- **Re-runs Required**: Count of failed submissions requiring manual retry.
- **Detection Incidents**: Count of IP bans, CAPTCHAs, rate limits encountered.
- **Average Runtime**: Strategy execution time (target: within baselines in section IV).

### Per-Team Metrics
- **Total Backlinks Generated**: Sum across all strategies.
- **Blockers & Dependencies**: List of external blockers (Yelp email verification, CAPTCHA detection, etc.).
- **Code Quality Score**: % of functions with docstrings, type hints, error handling.
- **Efficiency Improvement**: % reduction in runtime vs. previous week (via parallelization, optimizations).

### Quarterly Goals
- Success rate: 75% → 85%.
- Runtime: 60 min → 40 min (parallelization).
- Detection incidents: ≤2/month.
- Code quality: 100% strategy functions documented.

## VII. Challenge Questions — Ask These Before Approving Changes

### When reviewing ANY code change:

1. **Detection Risk**
   - "Does this change add, remove, or weaken rate limiting or randomization?" (If yes: CRITICAL review required)
   - "Are user-agents and referers rotated in every HTTP request?"
   - "Does the browser session pose re-identification risk?" (e.g., cookies leaking across campaigns)

2. **Code Quality**
   - "Does this function have a docstring explaining what it does?"
   - "Are errors caught specifically (not bare `except`)? Do they log context?"
   - "Is there duplicated logic that should be in `core/`?"

3. **Performance & Efficiency**
   - "Can this operation be parallelized (e.g., 2 strategies run simultaneously)?"
   - "Is this reading the CSV log on every submission, or pre-loaded?"
   - "Does this add ≥10% overhead to strategy runtime? If so, why?"

4. **Output Quality & Auditing**
   - "Is this submission logged to CSV with all required fields (timestamp, status, URL, notes)?"
   - "If this fails, is the reason logged explicitly (rate limit, auth, duplicate)?"
   - "Can we explain every row in CSV log?"

5. **Team Accountability**
   - "Who owns this code? Is ownership clear?"
   - "Can we measure success of this strategy objectively (success rate, runtime, blockers)?"
   - "Are blockers flagged so the team knows what's missing?"
